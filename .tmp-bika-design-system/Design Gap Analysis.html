03 — Dashboard

Codebase only

  • 6-KPI strip: Revenue, Received, Total dues, Today count, Pencils, Conflicts — all flat numbers, no trends
  • 14-day cash inflow bar sparkline (standalone chart below KPIs)
  • Enquiry funnel — horizontal bar chart by stage
  • Today's events table — time, function, customer, hall, guests, total, status
  • Alerts panel — conflicts, pencil expirations, overdue balances with left-border callouts
  • All data wired to live mock data (detectConflicts, bookingTotal, etc.)

Redesign only / improvements

  • 5 KPI tiles with inline sparkline + trend delta arrow (↑ / ↓) per tile
  • 12-month revenue bar chart (trailing year) with month labels and INR Lakh scale
  • Hall utilization donut chart + legend grid — visual venue mix at a glance
  • Upcoming events table (next 7 days) — same columns as codebase today-table
  • Insights panel — 4 cards with good/warn/bad/info kind badges and icons
  • Live activity feed — real-time event stream (who did what to which booking)
  • Today's checklist widget — 7 tasks, done/undone toggle, assignee labels
  • Date-range tab bar (Today / This week / 30 days / Quarter)
  • Export button and New Booking CTA in page header

The redesign dashboard is substantially richer. The codebase dashboard prioritises operational status (conflicts, pencils, dues). The redesign adds trend awareness (sparklines, 12m chart), strategic context (donut utilization, insights), and team coordination (checklist, live feed). Both are valid but serve different roles — consider merging the conflict/alert logic from the codebase into the redesign's alert section.

06 — Remaining Screens

Enquiries

kanban pipeline MATCH

Both implement a 5-column kanban: Lead / Quotation / Pencil / Won / Lost. Column headers show stage name, item count, and aggregated pipeline value. Cards display customer name, function type, date, guest count, estimated value, and halls. Implementation is nearly identical — the redesign uses the same semantic colour tokens per stage.

Customers

master-detail CRM MATCH

Both: 288px list sidebar with search, detail pane with contact fields (phone, alt phone, email, city, community, DOB, anniversary, occupation, company, GST, PAN), referral chain section, and full booking history table. The codebase adds star rating + visit count in the detail header. VIP badge rendered identically (pencil-colour border tag).

Payments

global ledger MATCH

Both: flat table of all payment transactions across all bookings — Date, Booking ID, Customer, Method, Reference, Received by, Amount. Sorted by date descending. Total shown in header. No meaningful difference.

Venues

hall catalogue MATCH

Both: venue-grouped hall tables with Hall name, Floor, Capacity, Floating capacity, Base price, Bookings (30d), Revenue (30d). Codebase version is wired to live mock data calculations.

Menu

Both have 4 tabs. Codebase has Items (with Veg/NV badge), Templates (menu packs with rate + setup cost), Ingredients (stock levels), Vendors (contact + items + last order). Redesign screen exists with equivalent content.

Activity Log

audit trail CODEBASE ONLY

Codebase has a dedicated Activity route (activity.tsx) — a full audit log table with When (datetime), User, Action verb, Target booking ID, Function name, and IP address. The redesign surfaces activity only as a live feed widget on the Dashboard, not as a dedicated browsable log screen.

Reports

analytics MINOR

Codebase: 2 horizontal bar charts — Revenue by function type, Hall utilization by hours. Both wired to live data. Simple and functional. Redesign: donut chart + insights panel on Dashboard partially covers this; a dedicated Reports screen is present but not materially different.

Settings

Both: tabbed settings with Profile, Users (stub), Roles (stub), Integrations (Google Calendar connected status), Theme toggle. Left-rail tab navigation in codebase; redesign uses similar pattern.

09 — Implementation Priority Matrix

What needs to be resolved before the redesign can be handed off to engineering as a complete spec for eventflow-ops.

P1 — Resolve before handoff
Nav: sidebar vs top-nav Structural decision affects every screen's available width and scrolling behaviour. Pick one — the redesign's sidebar is the better choice for this density of nav items.
Quotation colour: violet vs blue The conflict with the codebase's violet and redesign's blue must be resolved. Both read differently against the new teal accent.
Status badge style: border-only vs filled The redesign filled badges are more legible. The codebase statusToken pattern needs to be updated to match the redesign's --st-X-bg/fg/dot var system.
Typeface: IBM Plex vs Inter Tight Only one can ship. Inter Tight aligns better with the redesign's warmer, slightly more editorial tone. IBM Plex is more utilitarian/IBM-coded.
P2 — Bring into redesign
Conflict detection + pulse animation The pulse-conflict CSS animation and detectConflicts() logic are production-grade and not in the redesign calendar. Must be carried over.
Hall occupancy rail The per-hall mini occupancy bars in the calendar left rail are genuinely useful for operators. The redesign's filter bar doesn't include this.
Pencil hatch pattern The diagonal stripe on pencil-status events in the HallBoard is a strong visual cue. Decide: keep hatch or migrate to filled-badge approach consistently.
Version history tab in Bookings The audit trail (who edited, when, which version) is a real operational need. The redesign bookings sheet needs this tab.
Activity Log as dedicated screen The codebase has activity.tsx (full audit log with IP). The redesign only surfaces this as a live feed widget — the full log screen needs to be in the nav.
P3 — Nice to have / later
Google Calendar source visual Dashed-border event blocks for Google-sourced bookings in the codebase. Low visibility but useful for ops clarity.
Live clock in top bar The Mumbai real-time clock in the codebase is a small but appreciated ops-context detail — useful for staff knowing local time during events.
Density system — adopt in codebase The redesign's 3-tier density is a real feature. Implementing the CSS custom property approach in the codebase would let it serve both tablet and desktop use cases.
KPI sparklines on dashboard The redesign's sparkline + delta approach is a clear improvement over flat numbers. Worth porting the Sparkline component.
Hall utilization donut chart Nice visual addition on dashboard. Can co-exist with the existing funnel bars or replace them.

10 — Overall Verdict

The redesign is a strong visual and UX upgrade — the teal palette, sidebar nav, filled status badges, density system, and richer dashboard are all improvements over the codebase's current state. However, the redesign under-specifies the calendar (the most critical screen for daily ops) and is missing several production features that are already built in the codebase: conflict detection, hall occupancy rail, pencil countdown in event blocks, version history, and the activity log. The safest implementation path is to adopt the redesign's tokens + layout as the base and back-port the codebase's calendar and booking-detail feature depth on top of it.

Take from redesign → codebase

  • Warm stone palette + teal accent (replace zinc + blue)
  • Inter Tight + JetBrains Mono (replace IBM Plex pair)
  • Left collapsible sidebar with grouped nav + user footer
  • Filled status badges with --st-X-bg/fg/dot system
  • Density CSS var system (compact/balanced/comfy)
  • KPI sparklines + trend deltas on dashboard
  • Live activity feed (createActivityFeed pattern)
  • Gentle radius scale (4/6/8px vs current 2px)
  • Donut chart for hall utilization on dashboard
  • Insights + checklist widgets on dashboard

Keep from codebase → must carry over

  • detectConflicts() + pulse-conflict animation on HallBoard
  • Hall occupancy mini-bars in calendar left rail
  • Pencil expiry countdown inside event blocks
  • Google Calendar source distinction (dashed border + cyan token)
  • BookingDrawer with full money stack + payment ledger
  • Bookings detail: 6 tabs including Versions audit trail
  • Activity log as dedicated full-page route (not just live feed)
  • Real-time clock + Mumbai timezone label in top bar
  • statusToken() logic — update to output bg/fg/dot vars instead
  • Booking list: 420px master panel remains better than full-width table for this density of metadata per booking
